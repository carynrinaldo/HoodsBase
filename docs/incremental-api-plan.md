# Incremental API Exploration Plan

## Approach

Rather than building the full ServiceTrade sync system upfront, we're taking an incremental, discovery-driven approach. We work inside a Docker container, interact with the API one endpoint at a time, and let the Dockerfile and sync code grow from what actually works.

## Why

- The ServiceTrade API documentation leaves gaps — we need to see real responses to understand the data shapes, pagination behavior, and edge cases.
- Building incrementally means every piece of code is validated against the live API before we move on.
- The Dockerfile ends up containing exactly what's needed and nothing extra.

## Steps

### 1. Start with a minimal container — Done

- Base image: `python:3.12-slim`
- Initial dependencies: `requests` (for API calls), `sqlite3` (built-in)
- Bind-mount the project directory so we can edit files from outside the container
- Container name: `safehoods-dev`

### 2. Authenticate — Done

- Session auth chosen over OAuth2 — simpler, works well for a sync script
- Token persisted to `.session_token` file (gitignored) for reuse across runs
- Auth script (`sync/auth.py`) checks saved token first, only logs in if expired
- See `docs/auth-reference.md` for full details

**Discoveries:**
- `resourceBalanceMs: 60000` in response `meta.stats` — 60-second rate limit budget per minute (architecture doc previously said 30s)
- `eulaNeeded: true` in auth response — not blocking API access currently
- User account is admin type (`isTech: false`, `isHelper: false`)

### 3. Explore each resource, build mappings and context — Done

For each resource in priority order:
1. Run `python system/rebuild_mappings.py --only <endpoint> --verbose` — auto-generates `schema/mappings.yml` entry (fields, types, pagination)
2. Consult the relevant section of `docs/APIdocumentation.pdf` for business context
3. Add a `resource_overrides` entry in `system/api_knowledge.yml` with description and field rules

`schema/context.yml` is auto-generated by `rebuild_context.py` from `mappings.yml` + `api_knowledge.yml` — it is not hand-edited.

**Completed:** All 18 resources — serviceline, company, location, contact, job, appointment, invoice, invoiceitem, quote, quoteitem, asset, deficiency, servicerecurrence, user, region, tag, taxrate, paymentterms

**Key discoveries:**
- Response records are nested under `data.<plural>` (e.g. `data.servicelines`, `data.companies`, `data.locations`, `data.contacts`)
- Pagination info in response: `data.totalPages`, `data.page`, `data.totalCount`, `data.pageSize`
- Some resources fit in one page (serviceline: 11 records, company: 557 records, job: 1,179 records, quote: 627 records), others paginate (location: 640/64pp, contact: ~900/9pp, appointment: ~8,000/4pp, invoice: ~5,500/3pp)
- Boolean API fields need `db_type: integer` override for SQLite
- camelCase API names get snake_case `db_column` renames
- Nested address objects use `flatten` directive to become `address_*` columns
- FK objects (company, primaryContact) use `extract_key: id` to store just the ID
- Arrays of related objects (tags, locations, companies) are skipped — handled via join tables
- Contact has deprecated `type` (singular) and `company`/`location` (singular) fields — skipped in favor of `types` array and `companies`/`locations` arrays
- Job API defaults to `status=scheduled` unless another status is specified
- Invoice items are fully detailed in the parent invoice response (`fetch_mode: inline`)
- Quote items are sparse in the parent quote response (only id/uri/description) — full detail requires separate call to `/quote/{id}/item` (`fetch_mode: parent_endpoint`)
- Quote money fields (`subtotal`, `taxAmount`, `totalPrice`) are returned as formatted strings (e.g. "9,250.00") unlike invoice where they're numeric — `subtotal` handled by global `type_corrections`, others by resource overrides
- Child/embedded resources (invoiceitem, quoteitem) use `api_endpoint: embedded`, `parent_resource`, `parent_id_field`, and `fetch_mode` annotations in mappings.yml
- rebuild_mappings.py auto-detects child items and crawls `/{endpoint}/{id}/item` to generate companion mappings
- Asset endpoint requires mandatory filter params (`updatedAfter`) — returns 405 without them. Annotated with `required_params` in mappings.yml
- Asset `properties` field is flexible JSON varying by asset type (hood dimensions, filter sizes, fan specs)
- Service recurrence endpoint is slow (~17s response time), 4,415 records
- Region and tax_rate endpoints return 0 records — included as static resources for future use
- User endpoint returns 16 records with role booleans (isTech, isHelper, isSales)

### 4. Generate schema from pipeline — Done

`python system/rebuild_all.py` runs the full pipeline: `rebuild_mappings.py` → `rebuild_context.py` → `generate_schema.py`. The final stage reads `mappings.yml` + `context.yml` and produces `schema/schema.sql`. Features:
- Applies all context.yml rules: `db_column` renames, `db_type` overrides, `flatten`, `extract_key`, `skip`
- Token-optimized for MCP system prompt (~3,300 tokens vs ~10,000 before optimization)
- Inline comments only where column name alone isn't enough (enums, non-obvious semantics) — uses `prompt_comment` field from context.yml
- Conventions documented once in header: `_id` = FK, `_dt` = timestamp companion, JSON TEXT columns, INTEGER booleans
- `_dt` companion columns omitted from schema output (convention in header; sync.py creates them independently)
- Empty tables skipped (region, tax_rate with 0 records/0 fields)
- `TABLE_NAME_MAP` handles compound names (`invoiceitem` → `invoice_item`, `quoteitem` → `quote_item`, `servicerecurrence` → `service_recurrence`, `paymentterms` → `payment_terms`, `taxrate` → `tax_rate`)
- Appends admin tables (`sync_status`, `sync_log`) at the end
- Run: `docker exec safehoods-dev python /app/schema/generate_schema.py`

Current output: 18 tables (3 static + 13 core + 2 admin)

### 5. Build the sync script — Done

`sync/sync.py` — single generic loop, no per-resource Python code. Driven by `mappings.yml` + `context.yml`.

**Features:**
- CLI: `python sync/sync.py [resource] [--full]` — sync all, one resource, or force full re-pull
- Static resources skipped if table already populated (unless explicitly requested or `--full`)
- Incremental sync via `updatedAfter={last_synced_at}` from `sync_status` table
- Budget-aware throttling: monitors `resourceBalanceMs`, pauses below 10,000ms threshold
- Child item auto-detection: if inline items have >4 keys, extract inline (invoice items, 0 API calls); otherwise call per-parent API (quote items)
- Verbose console logging with per-resource progress and summary table
- Graceful partial failure: one resource failing doesn't stop others

**Discoveries during testing:**
- Job API returns only `status=scheduled` by default — must pass `status=all` to get all 6,081 jobs (vs 1,179 with default)
- Invoice items fully embedded in parent response (16 fields per item) — inline extraction takes 3.7s for 6,549 items
- Quote items sparse inline (only id/uri/description) — requires 627 individual API calls, ~102s total
- Full backfill: ~90-100 seconds for all 16 resources (~34,000 records)
- Rate limiting never triggered during testing — budget stayed above 57,991ms out of 60,000ms
- `_dt` companion columns generated by transform but silently filtered during upsert (schema tables don't have them, by design)
- 1 contact record out of 893 fetched was filtered during upsert (no valid columns after transform) — minor edge case

## Resource Priority Order

Based on the architecture doc, suggested order for API exploration:

1. **Auth** — get connected
2. **service_line** — small, static, simple shape
3. **company** — core entity, likely straightforward
4. **location** — nested under company
5. **contact** — associated with company/location
6. **job** — central entity, likely the most complex
7. **appointment** — tied to jobs
8. **invoice** / **invoice_item** — revenue data the CFO cares about most
9. **quote** / **quote_item** — conversion tracking
10. **asset** / **deficiency** — compliance tracking
11. **service_recurrence** — scheduling (critical but potentially complex)
12. **user** — technician data
13. **Remaining reference tables** — region, tag, tax_rate, payment_terms

## Architecture Decision: Three-Stage Pipeline

The API-to-database relationship is captured in a pipeline with one human-edited input and three auto-generated outputs:

**`system/api_knowledge.yml`** — the human-authored source of truth:
- Global rules: camelCase→snake_case, boolean→integer, FK object extraction, address flattening
- `timestamp_fields` — integer fields that are Unix timestamps
- `type_corrections` — fields where API type inference is unreliable (money fields as integer, sort fields as text)
- `is_prefix_renames` — boolean fields needing `is_` prefix
- `enums` — status/type values for inline SQL comments
- `resource_overrides` — per-resource field rules, descriptions, skips

**`schema/mappings.yml`** — structural, auto-generated by `system/rebuild_mappings.py`:
- Field names, inferred types, nested object keys
- Pagination metadata (total_pages, total_count, page_size)
- Static vs. dynamic resource classification (via `static_resources` list)
- Child resource annotations: `api_endpoint: embedded`, `parent_resource`, `parent_id_field`, `fetch_mode`
- Drives the sync script — a generic loop rather than per-table Python functions

**`schema/context.yml`** — auto-generated by `system/rebuild_context.py` from `mappings.yml` + `api_knowledge.yml`:
- Table descriptions (become single-line SQL comments)
- `prompt_comment` — terse inline comments only where column name isn't self-explanatory (enums, non-obvious semantics)
- `db_column` renames (camelCase → snake_case, semantic renames)
- `db_type` overrides (boolean → integer for SQLite, timestamp, type corrections)
- `flatten` directives (nested address objects → flat columns)
- `extract_key` directives (FK objects → store just the id)
- `skip` directives (arrays handled via join tables, deprecated fields)

Together they produce `schema/schema.sql` and drive the sync script's field transformations. See `system/README.md` for full pipeline documentation.
